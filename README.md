# MK EDA

## Getting started

### Первоначальная установка и настройка

Для работы над проектом требуется:
1. Python версии для Windows == 3.9, для остальных ОС >= 3.9.
2. При работе в Windows, если в системе установлено несколько версий Python, может потребоваться выполнить команду `poetry env use python3.9` для выбора правильной версии интерпретатора.
3. Установить [Poetry](https://python-poetry.org/docs/#installation) для работы с вирутальным окружением.
4. Выполнить команду `poetry install` в корне репозитория для создания виртуального окружения и установки необходимых пакетов.
5. Выполнить команду `poetry run pre-commit install` в корне репозитория для установки прекоммитных хуков, выполняющих проверки на соответствие стилю кодирования и автоформатирование кода.
6. Если вы работаете в Visual Studio Code, то полезно установить и настроить расширения Flake8 и Pylance для подсветки несоответствий pep8 и ошибок типизации. При работе в IDE следует выбрать интерпретатор Python из виртуального окружения, созданного в пункте 2. В `.vscode/settings.json` рекомендуется прописать следующие настройки:
```json
{
    "flake8.args": ["--config=setup.cfg"],
    "python.analysis.typeCheckingMode": "strict",
    "python.analysis.diagnosticMode": "workspace"
}
```

### Структура проекта

1. Весь код содержится в директории `mk_eda` и состоит из иерархии библиотечных модулей (или библиотек), скриптов и исполняемых модулей.
2. Библиотекой является некоторый самостоятельный набор функций, классов, данных и др., который может использоваться в других частях проекта. В директории библиотечного модуля находится скрипт `__init__.py` (чаще всего это пустой файл).
3. В директории `mk_eda/scripts` содержатся скрипты, т.е. небольшие программы, помещающиеся в один файл и не требующие разделения на отдельные файлы или модули, выноса части функциональности в библиотеки.
4. В директории `mk_eda/tools` содержатся тулы в виде исполняемых модулей. Данный вариант оформления подходит для программ, которые из-за своего объёма требуют разделения на несколько файлов, но при этом данные файлы не подходят для выноса в библиотеку.
5. Тесты размещаются в директорях `tests`, лежащих в директории тестируемого модуля. Имя файла теста должно быть образовано от имени файла тестируемого скрипта, например, для скрипта `example.py` файл с тестом будет называться `test_example.py`. Аналогичные правила действуют при написании юнит-тестов на функции, например, для функции `foo` функция теста будет называться `test_foo`.

### Запуск скриптов и модулей

1. Для запуска скриптов можно пользоваться командой вида `poetry run python <путь к скрипту>` (например, `poetry run python ./mk_eda/scripts/example.py -s abc -n 3`).
2. Для запуска модулей, содержащих скрипт `__main__.py` можно пользоваться командой вида `poetry run python -m <путь к модулю>` (например, `poetry run python -m mk_eda.tools.example_tool -s abc -n 3`). Путь к модулю может быть также относительным (например, можно запустить `poetry run python -m example_tool -s abc -n 3`, находясь в директории `<корень проекта>/mk_eda/tools`).
3. Также при наличии установленных всех необходимых пакетов можно запускать скрипты и модули без использования виртуального окружения. При этом в некоторых случаях может понадобиться установить переменную окружения `PYTHONPATH=<корень проекта>` (например, `python -m mk_eda.tools.example_tool -s abc -n 3` или `PYTHONPATH=. python ./mk_eda/scripts/example.py -s abc -n 3`).

### Запуск тестов и CI

1. Для запуска всех тестов в проекте можно выполнить команду `poetry run pytest`. Для запуска тестов на конкретный модуль можно указать путь к нему (например, `poetry run pytest mk_eda/example_lib`).
2. На каждый push в GitLab запускается CI. Он состоит из двух стадий: lint и test (см. конфигурационный файл `.gitlab-ci.yml`). На стадии `lint` запускаются те же самые проверки, что и в прекоммитном хуке, только для всех файлов проекта, а не только для внесённых изменений. На стадии `test` выполняется запуск тестов при помощи `pytest`.
3. Для внесения изменений в главную ветку `main` необходимо прохождение CI.
4. Запустить прекоммитные проверки для всех файлов проекта, а не только для изменённых, можно в т.ч. вручную, выполнив команду `poetry run pre-commit run --all-files`. При этом в некоторых случаях важно следить, не делает ли автоформатирование лишние действия (например, оно может изменять автосгенерированные скрипты, которые следует добавить в исключения).

### Обновление прекоммитных хуков

В файле конфигурации `.pre-commit-config.yaml` содержатся фиксированные версии пакетов. Для их обновления можно выполнить команду `poetry run pre-commit autoupdate`, после чего закоммитить изменения в файле конфигурации.

## Общие технические моменты

Предлагается для описания инверсных графов использовать очень простой формат описания схемы на основе JSON.

UPDATE:: По просьбе студентов формат был обновлен.

Схема - это словарь из трех записей:
1. Список имен входов (inputs).
2. Список имен выходов (outputs).
3. Сама схема в виде таблицы уникальных вершин (nodes).


Таблица уникальных вершин - это словарь, где каждая запись состоит из имени вершины (ключ) и списка (синатура), содержащего следующие элементы: название функции, реализуемой в вершине, и описание исходящих из вершины ребер (название вершины куда идет ребро и является ли оно прямым или инверсным (0 - инверсное, 1 - прямое)).

Описание AIG для конъюнкции выглядит так:
{'inputs':['x1','x2','x3'], 'outputs':['f'], 'nodes': {'y1':['and','x1',1,'x2',1], 'f':['and','y1',1,'x3',1]}}

Такой формат позволит нам легко синхронизировать различные задания в проекте. Формат является расширением формата, предложенного изначально.

Замечание: в качестве вершины, куда идет ребро может выступать константа 0 или 1. В этом случае в векторе вместо имени вершины просто указывается либо 0 или 1.

## Ошибки
### Не устанавливается pyeda
При запуске `poetry install` установка `pyeda` завершается с результатом `Fail` - не компилируется `espresso`.

Решение - запуск `poetry install` с параметром для gcc: `CFLAGS="-Wno-error=incompatible-pointer-types" poetry install`.
